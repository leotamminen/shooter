# shooter

Lightweight browser FPS, box-based levels, data-driven content. Three.js/Vite/TS.

## Architecture rules (non-negotiable)

- `core/` never references `content/` or `modes/` directly — it only consumes typed interfaces.
- All game content (weapons, enemies, maps, sounds) lives in `content/*.ts` as typed data, never hardcoded in logic.
- New game modes are built hardcoded first; a `GameMode` interface is only extracted once a second mode proves the shape is right.
- No physics engine — manual collision only; player is represented as a radius/circle (not a box), since later systems (pickup range, melee range) will want a "distance from player" concept anyway.
- Single-responsibility per file: one class/system per file, file name matches it; split by responsibility, not by line count — a file mixing unrelated concerns must be split regardless of length; if a file's job needs "and" to describe, it's two files.
- Shared/reusable logic (math helpers, raycasting, state machines) goes in `core/utils/`, never duplicated inline.

## Workflow rule

`git init` at project start if not already a repo. Commit at the end of every checkpoint with a message naming which checkpoint was completed. This applies to every future session, not just this one.

It owns a wholly separate `THREE.Scene`/`THREE.PerspectiveCamera` pair rather than sharing the main scene: the weapon mesh is a direct child of this second camera at a fixed local offset (superseded at checkpoint 14 — see below; the offset itself is no longer fixed, but the rotation-cancellation property described next depends only on being a child of the camera, not on the offset's value staying constant, so it's unaffected), which rigidly fixes it in that camera's view regardless of the camera's own rotation (a child mesh's view-space transform is always just its local offset relative to the camera rendering it — the camera's own world rotation/position always cancels out exactly in that math), so it stays put on screen with no per-frame rotation sync needed. `main.ts` renders it as a second pass (`render()`) after the main scene — `renderer.clearDepth()` plus a `0.01` near-plane camera guarantees the weapon draws in front of world geometry no matter how close the player stands to a wall, which a naive single-pass render sharing the main depth buffer could not guarantee. This call is gated in `main.ts`'s `animate()` on `gameState.playerState === "alive"`, the same pattern already used for `gameMode.update()` — `WeaponViewmodel` itself never imports `GameState` and makes no gameplay decisions, keeping it a pure rendering system per this checkpoint's own requirement. The mesh's appearance does not vary by equipped weapon yet — at checkpoint 13, only one weapon (`pistol`) existed in `content/weapons.ts`, so there was nothing to differentiate (superseded at checkpoint 15 — MAC-10 exists now, but building per-weapon appearance is still separate, undesigned scope; see the decisions log and future mechanics below). Checkpoint 13 fix (post-review): `WeaponViewmodel.updateOrientation(mainCamera)` — which copied the main camera's quaternion onto the viewmodel's own camera every frame — was removed as a provable no-op, along with its call site in `main.ts`'s `animate()`. Since the weapon mesh is a direct child of the very camera used to render it, that camera's world rotation always cancels out exactly in the view-space math regardless of what it's set to; copying a different quaternion onto it every frame changed nothing about what was rendered. The weapon staying fixed on screen was always solely a consequence of the child-of-camera parenting described above, never of this dead rotation sync.

`PlayerController` gained `getSpeed(): number` (checkpoint 14), measuring actual resolved horizontal displacement each frame — not the `MOVE_SPEED` constant, and not a discrete moving/idle flag — so it automatically reads lower while sliding along a wall and would automatically read higher under any future different movement speed (sprint, not built) with no code change needed anywhere that consumes it. `core/WeaponViewmodel.ts` gained a per-frame `update(speed, deltaTime)`, called from `main.ts`'s `animate()` right before `render()`, that composes the weapon mesh's local position from three sources every frame: the static `VIEWMODEL_CONFIG` base offset (checkpoint 13, unchanged), a continuous walk bob, and the sum of any active `addImpulse()` calls. The bob smooths the raw incoming speed internally (a per-second lerp) before deriving amplitude and frequency from it — this single smoothing step is what makes the bob both ramp in smoothly as the player starts moving and decay back to neutral smoothly when they stop, rather than needing a separate "stopping" case. `addImpulse(offset, decayTime)` is a new, generic mechanism: each call queues a temporary offset that decays linearly to zero over `decayTime` seconds, and multiple concurrent calls sum rather than overwrite each other. The *combined* magnitude of all currently-summed impulses is clamped to a fixed ceiling (`MAX_IMPULSE_MAGNITUDE`, chosen by visual verification against the current `VIEWMODEL_FOV`/near-plane/base offset, not derived from a formula and assumed correct) — the clamp applies to the total summed vector, never to an individual impulse and never as a limit on how many impulses can be active, so spamming or holding a rapid trigger still visibly stacks the offset up to the cap and holds there rather than being silently rejected. A hard clamp alone, tested in-browser, visibly read as the weapon freezing/getting stuck rather than straining under sustained fire, so `update()` also layers a small, fast, continuous jitter on top once the (pre-clamp) impulse magnitude is close to or at the cap — scaled by a cubed-eased proximity measure so it stays negligible for one normal impulse and only becomes clearly visible once several have stacked near or past the cap, additive to the clamp rather than a replacement for it. Verified in-browser via two temporary, uncommitted debug key bindings: overlapping impulses summed to a visibly larger combined offset than either alone and decayed back to neutral correctly; a single press alone showed no visible jitter; spamming/holding one of the two keys repeatedly showed the offset grow, then hold at the cap with a small continuous wobble instead of freezing static, reading as straining/kicking under sustained fire, with the mesh remaining fully on-screen throughout. That scaffolding was removed before this checkpoint's commit, and nothing in the shipped code calls `addImpulse()` yet (no fire-kick, reload-dip, damage-flinch, weapon-switch-dip, or melee-swing trigger exists — see future mechanics). Verified in-browser: standing still shows no bob (matching checkpoint 13 exactly), walking produces a smoothly increasing back-and-forth/up-down motion, releasing movement keys decays it back to neutral rather than cutting abruptly, and checkpoint 13's positioning/depth-clip/death-hide behavior are all otherwise unaffected.

Binding it as an arrow field at construction time makes this safe regardless of how far the reference travels before it's finally called.

**Room 3's door mechanism was corrected mid-checkpoint, before any of this shipped.** The original design had Room 2's terminal (`room2_terminal`) open Room 3's door directly the instant its `whoami` command ran successfully, via an `onCommand` callback on a second, otherwise-identical `Terminal` instance `main.ts` constructed just to carry it — which in turn required a `MapEntitySystem.getDoorMesh()` method and a `let mapEntitySystem` forward reference so that callback's closure could reach the door mesh before `mapEntitySystem` was constructed. Manual testing of that design found it broke this codebase's own established consistency: every other locked door (Room 1's, the vault's) is opened by the player interacting with a real `password_lock` and submitting input, proving they actually know the answer — a command auto-opening a door the moment it's typed skips that step entirely. It was corrected before shipping: Room 3's door now has a real `password_lock` (`campaign_lock_3`, `secretField: "username"`), checked against `room2_terminal`'s `username` exactly like every other lock in this codebase. `whoami` was demoted to a pure reveal — it prints the username with a copy button (the same accessibility treatment `cat` already gives Room 1's password and the vault pin) and has no side effects of its own. `core/HandsViewmodel.ts` is the new thing rendered instead of `WeaponViewmodel` whenever `hasActiveWeapon()` is false: bare procedural-box hands (a palm, four fanned/curled fingers, an opposable-looking thumb — the same "everything's a box" placeholder aesthetic as every other mesh in this project), built once as the right hand and mirrored for the left via `.clone()` + `scale.x = -1` rather than hand-built twice, sharing one generated skin-tone `CanvasTexture` (same technique as `ComputerMesh.ts`'s screen texture) across both. `core/utils/ImpulseOffset.ts` extracts the sum/clamp/near-cap-jitter impulse mechanism `WeaponViewmodel.addImpulse()` has owned since checkpoint 14, now that `HandsViewmodel` is a second real consumer with its own independently-tuned constants (a smaller max magnitude than the weapon's, since hands sit closer to center-screen). `WeaponViewmodel`'s own public `addImpulse(offset, decayTime)` signature is completely unchanged — it just delegates to an internal `ImpulseOffset` instance now — so every existing caller (the melee lunge, and this checkpoint's new fire-kick/swap-dip) needed no changes.

Three new interaction feedback moments were wired this checkpoint, all reusing `addImpulse()`/`ImpulseOffset` rather than new mechanisms: `InteractSystem` gained an `onSuccessfulInteract` callback (fired only after a real `userData.onInteract()` call, never on an E press that hits nothing) wired to a quick right-hand grab nudge, harmless to call even while the weapon viewmodel — not hands — is the one currently rendered; `Weapon.kickStrength` (a new optional field, unset on the melee-only knife) scales a per-shot fire-kick impulse fired from `WeaponSystem.fire()` on every successful shot, small for the M1911 and noticeably larger for the MAC-10, confirmed in-browser to stack visibly under sustained full-auto fire up to the same clamped ceiling checkpoint 14 already built — the exact "rapid-fire recoil stacking" scenario that checkpoint's own design notes anticipated but never had a real trigger to prove until now; and a weapon-swap dip fires from `pickupWeapon()`/the number-key and scroll-wheel switch handlers whenever the active slot changes to a different, already-occupied slot — deliberately excluded on the very first hands→weapon transition from an empty starting loadout, since that's a swap between two entirely different rendered objects, not a "dip" of the same persistent weapon model (see the decisions log for the exact guard).

Checkpoint 22 replaces the checkpoint-21-addendum's melee wiring entirely with a dedicated three-phase animation sequence, having found (during that addendum's own use) that a shared two-phase impulse trick played identically on whichever viewmodel happened to be showing couldn't read as a real choreographed attack — it could only ever nudge whatever was already on screen, never show a distinct performance. `core/utils/HandMesh.ts` is a new file, extracted from `HandsViewmodel.ts`'s inline geometry: `createHandMesh()` (palm/fingers/thumb, its own freshly-generated skin `CanvasTexture` per call rather than a shared material param) and `createKnifeMesh()` (blade + hilt, visible by default). `HandsViewmodel.ts` lost its permanent-but-hidden knife mesh and `setKnifeVisible()` entirely — the checkpoint-22 melee performer is never `HandsViewmodel`'s own right hand, so there's nothing left for that mesh to do — and both `HandsViewmodel`/`WeaponViewmodel` gained a `setSequencerOffset(offset)` method: an external offset an outside controller can hold nonzero to translate the mesh out of view, additive to (and independent of) each one's own sway/bob/impulse math, zeroed by `main.ts` whenever nothing is sequencing.

`core/MeleeViewmodel.ts` is the new dedicated performer: a single right hand plus knife (built from `utils/HandMesh.ts`, never the left hand and never a weapon model, regardless of what's equipped), copying `WeaponViewmodel`'s proven dual-scene/second-pass-depth-cleared render technique. `playStab()`/`playSwing()` start one of two explicit, timed tweens — not built on `ImpulseOffset` (see the decisions log for why a decaying/snapping impulse is the wrong shape for a choreographed performance). The stab eases out into a forward thrust for the first 35% of its duration, then eases in toward a quick pull-back-and-down for the rest; the swing is a single accelerating sweep off the left edge of view. `getDuration()` reports the currently-playing performance's total time so `MeleeSequencer` (below) can plan its own phase timing around it, read once right after `playStab()`/`playSwing()` is called.

`core/MeleeSequencer.ts` orchestrates four phases (`idle`/`retracting`/`performing`/`returning`) via `core/utils/StateMachine.ts` — the same generic, context-driven state machine `EnemyAI` already uses for its own idle/chase/attack states, reused here rather than a bespoke timer since the shape (a small set of named phases, each with its own enter/per-frame logic, transitions decided by the owning class's own `update()` rather than the state handlers) matched cleanly. `trigger(hitEnemy)` captures, once, which carrier (`WeaponSystem.hasActiveWeapon()`) was active and whether the swing connected — a second `trigger()` call while already mid-sequence is a no-op, belt-and-suspenders alongside `WeaponSystem`'s own pre-existing 1s melee cooldown, which is what actually prevents `V` from re-firing that fast in practice. `retracting`/`returning` each interpolate a `carrierOffset` to/from a fixed `HIDDEN_OFFSET` over ~0.12s; `performing` holds the carrier at `HIDDEN_OFFSET` (not rendered — see below) for exactly as long as `meleeViewmodel.getDuration()` reports, calling `playStab()` or `playSwing()` on entry depending on the captured `hitEnemy`. `getActiveLayer()` (`"carrier" | "melee"`) and `wasWeaponActive()` are what `main.ts`'s render loop reads every frame to decide which of the three viewmodels actually draws.

`WeaponSystem`'s `onMeleeAttack` callback signature widened to `(hitEnemy: boolean) => void` — `meleeAttack()` already computes this (whether its own short-range raycast hit something with an `onHit` handler) for its existing damage-application logic, so passing it through to the callback needed no new hit test, only threading an already-known value one step further. Damage/hit-detection timing itself is completely unchanged: `onHit?.(damage)` still fires synchronously inside `meleeAttack()`, before `onMeleeAttack()` is ever called — the melee sequence's retract/perform/return visuals run entirely independently of when damage actually lands, confirmed in-browser (a round-1 zombie died within one frame of pressing `V`, well before the sequence's `performing` phase even started).

`main.ts` wires `meleeSequencer.trigger(hitEnemy)` as `WeaponSystem`'s `onMeleeAttack` callback, using the same forward-declaration pattern already established for `gameMode` (a `let meleeSequencer: MeleeSequencer;` declared before `weaponSystem`'s construction, since the callback closure references it before it exists; assigned immediately after `weaponSystem` is constructed, the earliest point `meleeSequencer`'s own dependency — `weaponSystem.hasActiveWeapon()` — exists). `animate()` calls `meleeSequencer.update(deltaTime)` every alive frame alongside `gameMode.update()` (a no-op while idle), and the render branch now has three cases instead of two: while idle, unchanged from checkpoint 21 (`hasActiveWeapon()` alone picks weapon vs. hands, both sequencer offsets explicitly zeroed); while active, `wasWeaponActive()` (captured once at `trigger()`, not re-read live) picks which carrier's `setSequencerOffset()` gets `getCarrierOffset()`, and `getActiveLayer()` picks whether that carrier or `meleeViewmodel` is the one actually rendered that frame.

A real visibility bug was caught and fixed during this checkpoint's own headless-browser verification, before shipping: `MeleeViewmodel`'s first-guess `GUARD_POSITION`/`STAB_THRUST_POSITION` (modeled loosely after `HandsViewmodel`'s own hand offsets) projected below the visible screen area entirely — confirmed by calling `THREE.Vector3.project()` on the hand mesh through its own viewmodel camera and finding its NDC y-coordinate past `-1`. Both were corrected (shallower `y`, more negative `z`, re-deriving `STAB_THRUST_POSITION` by the same relative delta from the corrected guard pose) and reconfirmed on-screen via the same projection check plus screenshots showing the hand/knife clearly visible mid-performance. `STAB_PULLBACK_POSITION` was deliberately left alone — projecting off-screen-low there is the intended handoff cue to the carrier rising back into view from below in the `returning` phase, not a bug.

Verified in-browser (headless-browser polling of `MeleeSequencer`'s own state plus screenshots): triggering melee from an unarmed Campaign state shows the exact phase sequence `retracting(carrier)` → `performing(melee, isPerforming() true)` → `returning(carrier)` → `idle`, with `wasWeaponActive() === false` throughout and the knife visibly swinging off the left edge of the screen; triggering it armed, aimed at a zombie within melee range, shows `wasWeaponActive() === true` and the same phase sequence, with the zombie's health already gone (one-hit kill, round 1) within ~30ms of the `V` press — confirming damage timing is untouched by the new visual sequencing; three rapid `V` presses fired immediately after a trigger left the sequencer in exactly the one sequence already in progress, never restarting or corrupting it; the HUD's ammo/weapon-name display was unaffected throughout (still hidden in Campaign, still showing the correct weapon/ammo in Zombie Survival/Shooting Range); and a full regression pass across all three modes (fire, reload, melee trigger + rapid-press spam, weapon switch) on the default map showed no crashes and correct `hasActiveWeapon()`/sequencer-idle state at the end of each.

Checkpoint 23 adds a fourth weapon, AK-47 (`content/weapons.ts`, id `"ak47"`) — full-auto like the MAC-10 but heavier: `damage: 20`, `fireRate: 0.1` (slightly slower cadence than the MAC-10's `0.08`), `magSize: 30`, `startingReserveAmmo: 90`, `reloadTime: 2.5` (a full-rifle-reload feel, longer than either existing gun), `cost: 2500` (the priciest wall-buy so far), `kickStrength: 0.6` (between the M1911's `0.3` and the MAC-10's `0.8`) — all first-guess values, same "tune later by manual verification" approach every prior weapon's stats used. `fireSoundId` reuses `"pistol_fire"`, the same placeholder-sound reuse the MAC-10 established at checkpoint 15 — no new sound design this checkpoint.

More significantly, this is the first weapon with a real procedural viewmodel mesh instead of the generic gray box every weapon has rendered since checkpoint 13 — closing the "per-weapon viewmodel appearance" future-mechanics item that's been open since then. `core/utils/WeaponMesh.ts`'s `createAK47Mesh()` builds a five-box silhouette (receiver, barrel, magazine, handguard, charging handle) using the same procedural-box aesthetic and named-child-in-a-`Group` technique `ComputerMesh.ts` already established: three parts are named and exported (`AK47_BODY_NAME`, `AK47_MAGAZINE_NAME`, `AK47_BOLT_NAME`) so a future reload-animation checkpoint can `getObjectByName()` them individually (eject/insert the magazine, rack the charging handle) — the barrel and handguard have no anticipated individual use and stay unnamed, the same restraint `ComputerMesh.ts`'s own screen/keyboard children already showed. The magazine gets a small negative-X-axis rotation to read as the AK's distinctive curved mag despite being a plain box, and the handguard (not the magazine) is explicitly where a future support-hand checkpoint should grip, per the reference image the checkpoint was scoped against.

`WeaponViewmodel` is the one file with a real behavior change. `update()` gained an `activeWeapon: Weapon | null` parameter; internally, the single `weaponMesh` field was replaced with a wrapper `displayGroup` (the object actually repositioned every frame by the existing base+bob+impulse+sequencer math, unchanged) holding two permanent children built once at construction — the original generic box (`genericMesh`) and the new AK-47 mesh (`ak47Mesh`) — with only one ever `.visible` at a time. A cached `lastWeaponId` means the two `.visible` flags are only touched when the equipped weapon's id actually changes since the previous frame, not every frame regardless. Every weapon id other than `"ak47"` (M1911, MAC-10, and `null`) keeps showing the generic box, completely unchanged in geometry, material, or behavior — this checkpoint deliberately did not give M1911/MAC-10 their own meshes, matching its own explicit scope. `WeaponSystem` gained a plain `getActiveWeapon(): Weapon | null` accessor (mirrors the existing active slot's weapon, or `null` exactly when `hasActiveWeapon()` is `false`) — no new callback plumbing, `main.ts` just reads it once per frame and passes it straight into `weaponViewmodel.update()`.

One `wall_buy` entity linked to `"ak47"` was added to each of the three maps (`test-grid`, `corridors`, `campaign_room1`'s Room 1) purely for purchase testing — first-guess open-floor placements, same "expect retuning" caveat every other entity position in this project carries.

Verified in-browser (headless-browser purchase/state inspection across all three maps, plus render-hook frame capture for visual confirmation): AK-47 is purchasable on all three maps for exactly 2500 points, correctly filling an empty slot via the existing unmodified `pickupWeapon()`; `getActiveWeapon()`/`hasActiveWeapon()` and the `WeaponViewmodel`'s `ak47Mesh.visible`/`genericMesh.visible` flags flip correctly and exclusively on purchase and on every subsequent number-key switch, with zero lingering visibility overlap in either direction; firing produces the expected ammo drain at the `0.1` fire rate and fire-kick impulses scaled by exactly `0.6` (`{x:0, y:-0.012, z:0.03}`, confirmed via spied `addImpulse()` calls against the exact formula in `main.ts`); reloading restores a full 30-round magazine from reserve via the existing, unmodified reload logic; switching to/from the AK-47 fires the existing swap-dip exactly once per switch, same as any other weapon-to-weapon switch. A `THREE.Vector3.project()`-based check confirmed the AK-47 mesh and the generic box occupy the mathematically identical local position within `displayGroup` (both at local origin `(0,0,0)`), so they inherit identical on-screen scale/placement by construction — the specific screen framing used in this checkpoint's own debug-camera tests happened to place that shared position mostly below the visible frustum for both meshes equally (an existing `VIEWMODEL_CONFIG` characteristic dating to checkpoint 13/14, confirmed unchanged and not touched this checkpoint) rather than a checkpoint-23 regression. A full regression pass across all three modes (fire, reload, melee, weapon switching) showed no crashes and no change to M1911/MAC-10's appearance or behavior.

A same-checkpoint fix addressed two issues surfaced right after checkpoint 23 shipped. First, the AK-47's screen framing (confirmed above as "mathematically identical to the generic box, by construction") turned out to be the wrong goal once a real detailed mesh existed to look at — sharing the generic box's placement left far too little of the rifle's longer silhouette on screen. `WeaponViewmodel` gained a new `AK47_OFFSET` constant, applied only to `ak47Mesh`'s own local position within `displayGroup` (never to `VIEWMODEL_CONFIG`, which the generic box still reads unchanged), shifting it up and left; a `THREE.Vector3.project()` check confirmed the generic box's own projection is byte-identical to before the fix, and a screenshot confirmed clearly more of the AK-47's receiver/barrel/handguard is visible than the earlier "mostly off-frame" state. Second, `mac10_fire`/`ak47_single` — real `.wav` recordings the player added, with `content/weapons.ts` already pointing at them via `fireSoundId` — played silently, for the same reason every prior sound in this project needs two registrations, not one: a `content/sounds.ts` `SoundDef` entry and a `main.ts` preload call, both of which were missing for these two ids (`AudioSystem.play()` silently no-ops for anything never `load()`ed). Both were added, mirroring `pistol_fire`'s existing non-positional shape; `WeaponSystem.fire()` already called `audioSystem.play(weapon.fireSoundId)` generically since checkpoint 15, so no change was needed there. Verified via a spied `AudioSystem.play()`: equipping and firing the AK-47, MAC-10, and M1911 in turn produced exactly `"ak47_single"`, `"mac10_fire"`, and `"pistol_fire"` respectively, with no cross-contamination.

Checkpoint 24 adds a persistent support hand for two-handed weapons. `Weapon` gains two new optional fields: `hasSupportHand` (a real per-weapon boolean, set explicitly on all three ranged weapons rather than left to a default — `false` on the M1911, `true` on MAC-10/AK-47) and `supportHandOffset` (a local `{x,y,z}` within `WeaponViewmodel`'s `displayGroup`, meaningful only when `hasSupportHand` is `true`). MAC-10's offset sits forward and under the generic box; AK-47's is chosen to land roughly on `createAK47Mesh()`'s handguard, not its magazine, per the reference image this checkpoint (and checkpoint 23's own handguard placement) was scoped against — both are first-guess values, tuned by eye, same as every other weapon stat/offset in this project.

`WeaponViewmodel` gained a third permanent `displayGroup` child alongside `genericMesh`/`ak47Mesh`: a support-hand mesh built once at construction via `utils/HandMesh.ts`'s `createFistMesh()` (the same closed-fist shape `MeleeViewmodel` already uses, not the open-palm `createHandMesh()` — a fist gripping a handguard reads better than a flat hand) and mirrored into a left hand via `scale.x = -1`, the same technique `HandsViewmodel` already established for its own two hands. Its visibility and position are set in the exact same `weaponId !== lastWeaponId` check the checkpoint-23 AK-47 mesh swap already added — a separate decision from which weapon mesh is shown, keyed off `activeWeapon?.hasSupportHand`/`activeWeapon.supportHandOffset` rather than the AK-47 id check. Being a `displayGroup` child (the object that already receives every one of `WeaponViewmodel`'s existing transforms — base offset, walk bob, `ImpulseOffset`-summed impulses, and `MeleeSequencer`'s `setSequencerOffset()`) means the support hand inherits all of that motion automatically, with zero new per-frame code: no separate bob/impulse/sequencer wiring was written or needed for it.

Verified in-browser (headless-browser state inspection plus render-hook screenshots): the M1911 shows no support hand, matching pre-checkpoint behavior exactly; equipping MAC-10 or AK-47 (via the same `pickupWeapon()` a real wall-buy purchase calls) shows the hand at exactly its configured `supportHandOffset`, positioned near the handguard/foregrip rather than floating disconnected or overlapping the magazine; switching back to the M1911 hides it immediately with no stray leftover mesh; `displayGroup`'s own position was confirmed to change during a fire-kick (proving the support hand — a `displayGroup` child — rides along automatically, since Three.js's own parent-child transform composition guarantees this structurally rather than needing a runtime check); triggering melee correctly drove the sequencer through its full `retracting`/`performing`/`returning` cycle back to idle, with the support hand still correctly visible and positioned for the equipped weapon once the cycle completed. A full regression pass across all three modes showed no crashes, and `HandsViewmodel.ts`/`MeleeViewmodel.ts` were both confirmed completely untouched (`git diff --stat` empty for both), matching this checkpoint's explicit "`WeaponViewmodel` only" scope.

Checkpoint 25 gives the AK-47 a real multi-phase reload choreography, driven by checkpoint 24's support hand and checkpoint 23's named parts, while M1911/MAC-10 get a much simpler generic dip — all of it a purely visual/audio layer on top of the existing reload timer (`isReloading`/`reloadTimeRemaining`/ammo restore), none of which changed. `WeaponSystem` gained an `onReloadStart(weapon, emptyReload)` callback, called once from `startReload()`, with `emptyReload` (`currentAmmo === 0`) captured before `isReloading` even flips true and long before `finishReload()` ever touches ammo — a tactical reload (some ammo still chambered) never racks the bolt, since a round is already chambered; an empty reload (fired the mag dry) always does.

New `core/ReloadSequencer.ts` drives the AK-47's choreography via two percentage-of-`weapon.reloadTime` breakpoint tables (empty: tilt-out/mag-out/mag-rise/mag-in/bolt-pull/bolt-release/tilt-back; tactical: the same minus the two bolt phases) — every phase's duration is a percentage, never a hardcoded second value, so a future reload-speed perk scales the whole animation automatically just by changing the one number the underlying timer already reads. It exposes `getTiltAngle()`/`getMagazineOffset()`/`getBoltOffset()`/`getSupportHandOverride()`, each computed from the current phase and its own local (0-1) progress within that phase; magazine-rise and magazine-in are deliberately treated as one continuous eased return to rest (not two disjoint eases) so there's no visible kink where they meet, and the same technique joins bolt-release with tilt-back (or, for a tactical reload with no bolt phases, tilt-back alone fills that same "ease back to normal" role). `getSupportHandOverride()` returns an absolute local position (not a delta, unlike the other three getters) — `null` outside the phases it's meaningful for, in which case `WeaponViewmodel` falls back to the weapon's own static `supportHandOffset` from checkpoint 24; it tracks the magazine's animated position (plus a small grip adjustment) through the magazine phases, then the bolt's position through bolt-pull, then eases from wherever it last was toward the weapon's own resting `supportHandOffset` (captured once at `trigger()`, since `Weapon` already carries that field) through bolt-release/tilt-back.

Six new sounds (`content/sounds.ts` + `main.ts` preload, both steps for every one — the exact requirement the checkpoint-23 fix had to add retroactively for `mac10_fire`/`ak47_single`) are short synthesized placeholder tones (the same Node-generated-PCM technique as every other placeholder sound in this project), each played once by `ReloadSequencer` at the instant its own phase begins, tracked via a `lastPhaseName` guard so a phase's sound never repeats while elapsed time remains inside it. `core/utils/WeaponMesh.ts` gained two small exports, `AK47_MAGAZINE_REST_POSITION`/`AK47_BOLT_REST_POSITION` — the exact values already passed to each mesh's `.position.set()`, just given a name so `ReloadSequencer`'s offsets can be added on top rather than read back from a live (possibly already mid-animation) mesh.

`WeaponViewmodel` gained a `reloadSequencer` parameter on `update()`. For the AK-47 specifically, while `reloadSequencer.isActive()`, it applies `getTiltAngle()` to `displayGroup.rotation.z`, adds `getMagazineOffset()`/`getBoltOffset()` on top of the magazine/bolt's rest positions (retrieved once at construction via `getObjectByName()`), and overrides the support hand's position with `getSupportHandOverride()` when non-null, falling back to the static `supportHandOffset` otherwise — the AK-47 mesh itself is never swapped out or hidden during this, unlike `MeleeSequencer`'s carrier hand-off. Once the sequence ends (or a different weapon is equipped), the AK-47's tilt/magazine/bolt/support-hand are explicitly snapped back to their exact rest values rather than left at whatever `ReloadSequencer`'s own last active frame happened to compute — its own elapsed-time check can end the sequence a fraction of a frame before the easing curves mathematically reach their true endpoint, so this guarantees no residual drift lingers. M1911/MAC-10 skip all of this — `main.ts`'s `onReloadStart` branches on `weapon.id === "ak47"`; anything else calls `weaponViewmodel.addImpulse({x:0,y:-0.1,z:0}, weapon.reloadTime)` (a decay time equal to the weapon's own `reloadTime`, so it naturally eases back up over the whole reload with zero phase logic and automatically scales with any future reload-speed perk) plus `audioSystem.play("reload_generic")` once.

Verified in-browser (headless-browser property/timing inspection, a spied `AudioSystem.play()`/`WeaponViewmodel.addImpulse()`, and direct polling of every `ReloadSequencer` getter through a full reload cycle): a tactical reload (fired once, reloaded with 29/30 ammo) played exactly `mag_out`/`mag_rise`/`mag_in` with no bolt sounds; an empty reload (mag emptied fully) played the same three plus `bolt_pull`/`bolt_release`; both completed in ~2.5-2.6s wall-clock time, matching the AK-47's `reloadTime`; polling every getter across a full empty-reload cycle showed the tilt ramping to its max and back, the magazine offset peaking mid-out and easing back to exactly `(0,0,0)` by the end of mag-in, the bolt offset peaking mid-pull and easing to `(0,0,0)` by the end of bolt-release, and the support hand's override smoothly handing off from tracking the magazine to tracking the bolt to landing almost exactly on the configured `supportHandOffset` (`-0.08,-0.05,-0.35`) by the sequence's end; firing was confirmed blocked while `isReloading`, and switching weapons mid-reload correctly cancelled it (`isReloading` false, active weapon changed) via `WeaponSystem`'s existing, unmodified `switchToSlot()` logic; triggering melee mid-reload ran to completion on its own independent cooldown with no interference to the reload's own timer or final ammo restore; M1911's reload played exactly one `reload_generic` and one `addImpulse({x:0,y:-0.1,z:0}, 1.5)` call (matching the M1911's `reloadTime` exactly). A full regression pass across all three modes showed no crashes.

## Decisions log

- No physics engine: manual collision, chosen for full control and zero dependency weight.
- Hitscan only, no projectile physics, for simplicity.
- Player collision is radius-based, not box-based: simpler wall resolution, reusable later as a general distance-from-player check.
- Audio via Three.js native AudioListener/PositionalAudio, no external library.
- Weapon fire sound is non-positional (`SoundDef.positional: false`): heard the same regardless of player position since it's always the local player firing; positional audio is deferred to checkpoint 4 (zombie growls), the first sound that actually needs a 3D source.
- Placeholder fire sound is a synthesized WAV (short decaying sine click), generated programmatically since no real audio asset was available — flagged for replacement, not meant to ship.
- `MapLoader.loadMap()` exposes raw wall meshes (`walls: THREE.Mesh[]`) alongside `wallBoxes` (checkpoint 2): a `Box3` isn't raycastable, so `WeaponSystem` needed the actual scene objects to hit-test against. `InteractSystem` (checkpoint 3) reuses the same `walls` array for its own raycast, combined with whatever interactables are in range — one shared list of "things that can occlude or receive a ray," no duplicate wall representation.
- Interactable objects are tagged via `mesh.userData.interactable = true`, not by name/convention matching, so `InteractSystem`'s check is a plain property lookup regardless of what the object is named.
- Line-of-sight blocking for interact/hitscan comes for free from raycasting against a combined target list (walls + interactables) and taking the nearest hit — no separate occlusion check needed.
- `Weapon.name` added as a separate field from `Weapon.id` — id is the data lookup key, name is player-facing display text.
- HUD (`ui/HUD.ts`) is a plain DOM overlay, not part of the Three.js scene: absolutely positioned, `pointer-events: none` so it never steals clicks from the canvas (pointer lock, firing, interact all still work). It reads only from `GameState`; `WeaponSystem`/`InteractSystem` write their relevant fields to `GameState` every frame. This replaces the checkpoint-2/3 `console.log` observability, which has been removed now that the same information is visible on screen.
- The 1-second delay before showing "Press R to reload" is tracked inside `HUD.ts` itself (a local `emptySince` timestamp), not in `WeaponSystem` or `GameState` — it's presentation timing, not a gameplay rule, so it belongs with the thing that renders it.
- Damage sources use a generic `userData.onHit(damage)` callback on the target mesh, the same pattern as `userData.interactable` — `WeaponSystem` calls whatever hook exists on the object it hit and never imports `EnemyAI`. Any future damageable object (barrels, other enemies) plugs into `WeaponSystem` for free by setting this one field.
- The floating enemy health label (current/max, projected above the enemy's head via `camera.project()`) is a debug/test aid for verifying damage and state transitions, not a final UI choice — it should be replaced with a real health bar, hidden, or otherwise redesigned once the game is closer to presentable. Noted here so it isn't mistaken for a deliberate design decision later. Checkpoint 4.5 fixed a bug where it showed through walls (it's now hidden whenever a wall occludes the enemy or the enemy is behind the camera), but its status as a placeholder is unchanged.
- `core/utils/Health.ts` centralizes "clamp damage at 0, fire a callback exactly once on crossing to zero" as a plain function (`applyDamage(current, amount, onZero?) → number`), not a class — it doesn't own any state itself, the caller still stores the returned value. This is the same reuse reasoning as `Raycast.ts`/`StateMachine.ts`: both the enemy's own health (in `EnemyAI`) and the player's health (in `PlayerState`) route through it instead of duplicating "clamp and detect zero" twice.
- `GameState.playerState` is a string union (`"alive" | "dead"`), not a boolean, because a future `"downed"` value (perk-gated revive, not built yet) sits between them — a boolean would force a breaking type change later. See "Future mechanics" below.
- Player health/lifecycle logic lives in its own file, `core/PlayerState.ts`, rather than in `PlayerController` or `WeaponSystem`: those two are movement-only and firing-only per the single-responsibility rule, and "what happens when health hits zero" isn't either of those things. `PlayerController` and `WeaponSystem` only read `gameState.playerState` to decide whether to no-op; only `PlayerState.applyDamage()` (called by whatever deals the damage, e.g. `EnemyAI`) is allowed to change it.
- Score (`gameState.score`) is awarded from inside `EnemyAI.takeDamage()` — the same method that already routes through `applyDamage()` — rather than from `WeaponSystem`'s generic `onHit` call site: `WeaponSystem` fires the hook on anything with `userData.onHit` (currently only the zombie) and has no notion of "this hit was worth points," so scoring stays with the thing that knows it died, not the thing that pulled the trigger. +10 is applied unconditionally on every processed hit; +50 is applied inside the same `applyDamage()` call's `onZero` callback, so a killing shot stacks both in one `takeDamage()` invocation rather than the two being alternatives.
- **Superseded at checkpoint 10** (was: "`score` is a permanent total, never decreased; `pointsBalance` is the future spendable currency for wall-buys etc. They start identical (both incremented together via `GameState.addScore()`, a single mutator so they can't drift out of sync by accident) but will diverge once spending exists (checkpoint 6+, spending only decrements `pointsBalance`). Do not conflate them later."): spending now exists — `GameState.spendPoints()` is the one mutator that ever decrements `pointsBalance`, and `score` still never decreases. The two numbers can now genuinely diverge the first time a player spends, which is exactly why `ui/HUD.ts`'s live corner reads `pointsBalance` while the death panel still reads `score` (see the checkpoint-10 decisions below).
- **Superseded at checkpoint 4.8** (was: "`PlayerState.respawn()` deliberately does not touch `score`/`pointsBalance` — only checkpoint 7's round/mode logic should ever decide whether score resets, not the respawn action itself"): a run ends when the player dies, and pressing Respawn or Main Menu starts a new run, which resets `score`/`pointsBalance` to 0. `PlayerState.respawn()` itself is unchanged — it still only resets health/lifecycle (`playerHealth` back to `PLAYER_MAX_HEALTH`, `playerState` back to `"alive"`) and still never touches `score`/`pointsBalance` directly. The reset is orchestrated one level up, by `RunManager.startNewRun()`, which resets score before calling `respawn()`. Position reset remains a separate call to `PlayerController.setSpawn()` from `main.ts`, since `PlayerState` has no reason to know about camera/world position.
- `core/RunManager.ts` is a small registry (`registerResettable(fn)` + `startNewRun()`), not a god-object: it holds no per-type knowledge of weapons, enemies, or anything else. Systems register their own `reset()` closure with it once during their own construction — the same generic-hook pattern already used for `userData.onHit` and `Health.ts`'s `onZero` callback. `startNewRun()` calls every registered `reset()`, then `GameState.resetScore()`, then `PlayerState.respawn()` last, in that order.
- **Superseded at checkpoint 10** (was: "`GameState.resetScore()` was added alongside the existing `addScore()` mutator, for the same reason `addScore()` exists: `score` and `pointsBalance` must never drift out of sync, so both fields are only ever touched together through one of these two methods, never assigned directly by a caller."): `GameState.spendPoints()` is now a third sanctioned mutator, and it deliberately breaks the "never drift out of sync" invariant — that divergence was always the intended eventual payoff of splitting `score` (permanent) from `pointsBalance` (spendable), per the original checkpoint-4.6-era design. `addScore()`/`resetScore()` are unchanged and still always touch both fields together; `spendPoints()` is the one sanctioned exception, touching only `pointsBalance`. The "never assigned directly by a caller" part of the original rule still holds — every touch to either field still goes through one of these three named mutators, never a raw field assignment.
- **Superseded at checkpoint 7** (was: "`EnemyAI.die()` sets `mesh.visible = false` instead of removing the mesh from the scene graph, so `EnemyAI.reset()` (registered with `RunManager`) can bring the zombie back — full health, idle state, spawn position — without recreating geometry/material"): enemies are now spawned and destroyed per-round by `ZombieSurvival`, not persistent singletons, so there's no longer a scenario where a specific enemy instance needs to un-die. `EnemyAI` no longer registers directly with `RunManager` at all. A dead enemy is now fully `destroy()`ed — removed from the scene, geometry/material disposed, removed from `WeaponSystem`'s raycast targets, its `gameState.enemyHealth` entry deleted. `destroy()` is idempotent (guarded by the `dead` flag) and is called both by a natural kill (`onDeath()`, which also awards score and plays the death sound) and by `ZombieSurvival.resetRun()` forcibly clearing every still-alive instance on a new run (no score, no sound — it wasn't a kill).
- `core/utils/Raycast.ts`'s `cast()` now filters intersections down to `object.visible` ones, since three.js's `Raycaster.intersectObjects` does not skip invisible objects on its own. This is what actually excludes a dead (hidden) enemy from `WeaponSystem`'s hitscan, `EnemyAI`'s own line-of-sight check, and `HUD`'s occlusion check — all three get it for free from the one shared utility, with no per-caller "is this thing dead" check needed.
- `PlayerState` takes an optional `onDeath` callback (invoked from inside the same `applyDamage()` zero-crossing that flips `playerState` to `"dead"`) rather than importing `PlayerController` directly. `main.ts` wires it to `playerController.controls.unlock()` — releasing pointer lock is what makes the death panel's buttons clickable, but `PlayerState` has no business knowing about pointer-lock/DOM concerns, so it's handed in as a hook instead of reaching across.
- HUD's death panel is the one part of the HUD tree with `pointer-events: auto` (everything else stays `none`, per the checkpoint-3.5 decision) — it's only visible while dead, so enabling clicks on it never risks stealing input from gameplay.
- On death, `HUD` doesn't just let the death panel visually cover the crosshair/ammo/interact prompt — it actively hides the crosshair (`display: none`) and clears the other elements' text content, skipping their normal per-frame update methods entirely while `playerState !== "alive"`. This was an explicit requirement (elements must not render at all, not just be behind the death screen), and matters in practice because those elements have no background — without this they'd visibly show through/around the semi-transparent death panel.
- `core/utils/Lookup.ts`'s `findById<T extends { id: string }>(list, id)` is the one shared lookup for every content array (`WEAPONS`, `ENEMIES`, `SOUNDS`, `MAPS`) — same reuse rule as `Raycast.ts`/`StateMachine.ts`/`Health.ts`, so there's no per-content-type lookup function. It throws (naming the missing id) rather than returning `undefined`, since a silent `undefined` would surface later as a confusing null-reference error far from the actual mistake (a typo'd id in `main.ts`).
- Player spawn position moved from standalone `SPAWN_X`/`SPAWN_Z` constants in `main.ts` into `MapDef.entities` as a `{ type: "spawn", position: [...] }` entry (checkpoint 5), read back out via `MapLoader.getSpawnPosition()`, which throws if a map has no spawn entity rather than silently defaulting to the origin. This reuses `MapEntity`'s existing type (already had `"spawn"` in its union since checkpoint 1) instead of adding a second, parallel way to declare spawn data.
- `Weapon.startingReserveAmmo` was added so `WeaponSystem`'s starting/reset reserve ammo is weapon data, not a `main.ts` constant passed in alongside the `Weapon` object — this also meant dropping the separate `reserveAmmo` constructor parameter `WeaponSystem` used to take, since it would otherwise be a second, redundant way to say the same number.

- Walk bob (checkpoint 14) is driven by a continuous function of `PlayerController.getSpeed()` — itself measured from actual resolved displacement, not the `MOVE_SPEED` constant — rather than a lookup table keyed by a discrete movement-state enum (e.g. `"idle" | "walking"`). No such enum exists anywhere in this codebase and none was introduced. This is deliberate: a continuous formula means any future different movement speed (sprint, not built yet) automatically produces proportionally more bob the moment it exists, with zero changes to `WeaponViewmodel` or `PlayerController` — a state-keyed lookup table would need a new entry (and a decision about where sprint's threshold sits) every time a new speed tier was added.
- `WeaponViewmodel.addImpulse(offset, decayTime)` (checkpoint 14) is a generic, weapon/trigger-agnostic mechanism, deliberately built before any real trigger exists: each call queues an offset that decays linearly to zero over `decayTime` seconds, and `update()` sums every still-active impulse's contribution every frame. This is the intended integration point for fire-kick, reload-dip, damage-flinch, weapon-switch-dip, and melee-swing (none implemented this checkpoint — see future mechanics); building the summing mechanism generically now, rather than a single-slot "current kick offset" field, is what lets those effects overlap correctly later (e.g. a reload-dip still in progress when a shot fires) without a redesign.
- The bob's raw speed input is smoothed internally by `WeaponViewmodel` (a per-second lerp, `BOB_SPEED_SMOOTHING`) before being used to derive amplitude/frequency, rather than smoothing happening in `PlayerController` or being left unsmoothed. `PlayerController.getSpeed()` reports the true instantaneous measured speed (useful in its own right, unsmoothed, for any future consumer); the smoothing is purely a presentation concern of how the bob *looks*, so it belongs entirely inside the rendering-only `WeaponViewmodel`, not leaked into `PlayerController` or a third file.
- `MAX_IMPULSE_MAGNITUDE` (checkpoint 14) clamps the *combined* magnitude of all currently-summed active impulses, not any individual impulse's own offset and not how many impulses can be active at once — a rapid-fire weapon's recoil (not built yet) is exactly the shape of input this needs to handle correctly: many overlapping `addImpulse()` calls in quick succession, which should visibly stack the effect up to a fixed ceiling and hold there, not be silently capped by count (which would look like recoil "running out" arbitrarily) and not be left unclamped (which could push the mesh off-screen entirely, discovered as a gap after the base bob/impulse mechanism was already built and verified). The clamp preserves direction (scales the summed vector down proportionally when it exceeds the cap) rather than clipping each axis independently, so a diagonal impulse combination still decays toward the base position along a straight line rather than snapping onto a box boundary. The specific value (`0.15`) was chosen and confirmed by visual verification in-browser against the current `VIEWMODEL_FOV`/near-plane/base offset — spamming/holding the checkpoint-14 test impulse key showed the mesh staying fully on-screen at the cap — not derived from a frustum-geometry formula and assumed correct; revisit if `VIEWMODEL_FOV`, the base offset, or the near-plane ever change.
- A hard clamp on impulse magnitude, tested in-browser immediately after being built, visibly read as the weapon freezing/getting stuck at the ceiling rather than straining under sustained fire — this was caught by manual testing, not anticipated in the original design. Rather than replacing the clamp (the cap's actual value was already confirmed correct — the problem was the clamp holding perfectly static, not the ceiling being wrong), checkpoint 14 adds a small additive near-cap jitter: a fast sine wobble on the X/Y axes, scaled by a cubed-eased function of how close the (pre-clamp) impulse magnitude is to `MAX_IMPULSE_MAGNITUDE`. Cubed (not linear) easing was chosen specifically so the jitter stays negligible for one normal-sized impulse decaying on its own — the case the rest of this checkpoint already verified independently — and only becomes clearly visible once several impulses have stacked close to or past the cap, rather than a linear ramp that would introduce a small but noticeable wobble even on an ordinary single shot. The two axes use different phase multipliers (not both the same sine) deliberately, to avoid reading as a mechanical back-and-forth metronome and instead read as a small chaotic strain. `JITTER_FREQUENCY`/`JITTER_MAX_AMPLITUDE` were tuned by eye in-browser for a "straining/kicking under sustained fire" read.
- MAC-10's full-auto behavior (checkpoint 15) required zero new mechanics: `WeaponSystem.update()`'s existing fire-rate gate (fire whenever `firing && timeSinceLastShot >= weapon.fireRate`, for as long as mouse1 is held) already produces full-auto for any sufficiently small `fireRate` — this was true the moment `Weapon.fireRate` became per-weapon data at checkpoint 5, it just had no second weapon to demonstrate it until now. This is the payoff of the project's data-driven-content rule: a mechanically distinct-feeling weapon (full-auto vs. semi-auto) is purely a content addition, not a logic change.
- MAC-10's specific stat values (checkpoint 15) — `fireRate: 0.08`, `damage: 8`, `reloadTime: 1.2`, `cost: 1200` — are first-cut judgment calls, not derived from a formula or a balance model (this project has no economy/balance framework yet). `fireRate` was the one value actually tuned against manual testing (confirmed to read as clearly full-auto and clearly distinct from the pistol's `0.3`); the rest follow conventional SMG-vs-pistol tradeoffs (lower per-hit damage, quicker reload, higher cost) without a specific target number in mind. Revisit if a real balance pass ever becomes in scope.
- The weapon viewmodel's appearance remains generic regardless of equipped weapon (checkpoint 15 confirms checkpoint 13's deferral is still the right call, now that a second weapon actually exists to prove the gap): MAC-10 and the pistol render identically despite very different firing behavior. Per-weapon viewmodel appearance (shape/color/scale, or eventually a real MAC-10 model) remains future scope — see future mechanics; this checkpoint deliberately did not use the arrival of a second weapon as a reason to also build that.
- `WeaponSystem`'s slot-based inventory (checkpoint 15) uses one named `MAX_SLOTS` constant, currently `2`, rather than a bare `2` inlined wherever slot count/indexing matters — a future perk is planned to raise it to `3`, and the number-key handler (regex-matched over any digit, bounds-checked against `MAX_SLOTS`) and the scroll-wheel handler (cycles over however many slots are currently occupied, whatever that count is) were both written generically for this reason, so raising the constant alone is expected to require no other code change. Not verified end-to-end with an actual 3rd slot (no perk system exists to grant one) — this is a design intent, not a tested guarantee.
- `currentAmmo`/`reserveAmmo` are tracked per-slot, not globally (checkpoint 15) — switching away from and back to a weapon restores exactly the ammo it had when you left it, rather than either sharing one ammo pool across weapons (which would make no sense for different `magSize`/`startingReserveAmmo` weapons) or resetting on every switch (which would make switching a free reload exploit). `isReloading`/`reloadTimeRemaining`/`timeSinceLastShot` deliberately stayed single (not per-slot): switching cancels any in-progress reload of the weapon left behind rather than tracking independent reload state per weapon, matching how most FPS games handle a weapon swap mid-reload — this was a judgment call, not specified, and is the simplest defensible behavior given no other requirement was stated.
- The wall-buy's fill-empty-then-replace-active rule (checkpoint 15, `pickupWeapon()`) deliberately never touches a slot other than the empty one (if any) or the currently active one — buying a weapon while both slots are full always replaces whatever you're currently holding, never a different, unselected slot, even if that unselected slot holds a "worse" or duplicate weapon. This was spec-mandated, not a judgment call, but is recorded here because it's easy to misimplement as "replace the oldest slot" or "replace slot 0" instead.
- The main menu's Weapon selection no longer determines starting loadout (checkpoint 15) — **confirmed directly with the user** via an explicit clarifying question, since this is a real behavior change to a feature shipped at checkpoint 9 (load-time mode/map/weapon/enemy selection), not an incidental side effect. The alternative (menu selection still picks slot 0's starting weapon) was presented and explicitly not chosen. The Weapon selection group itself was left in place, unchanged, rather than removed or repurposed — see future mechanics for the two follow-up options (remove it, or give it real meaning again, e.g. choosing which weapon starts in slot 0 alongside a *guaranteed* M1911 in slot 1, or similar).
- The melee attack's `damage: 100` (checkpoint 16) is deliberately, exactly equal to `content/enemies.ts`'s zombie `EnemyDef.health` (100) — this is a load-bearing coupling between two numbers in two different files, not a coincidence, and **must not be independently "corrected" or rebalanced** without updating both together: `ZombieSurvival.healthForRound(round)` scales zombie max health as `enemyDef.health * round`, so as long as the melee attack's damage equals `enemyDef.health`, round N always takes exactly N melee hits to kill — the explicit design goal for this checkpoint. Changing either number alone (e.g. a future balance pass raising zombie base health without raising melee damage to match) silently breaks this property.
- Round-based zombie health scaling (checkpoint 16) uses the plain formula `enemyDef.health * round`, uncapped — no maximum round, no diminishing-returns curve. This intentionally lets kill-shot counts (for guns too, not just melee) rise without bound at high rounds, matching real CoD Zombies' own scaling. `ZombieSurvival`'s shared `enemyDef` object is never mutated to hold a round-specific value; `startRound()` computes a fresh local `number` each round and passes it explicitly into each `EnemyAI`'s constructor, keeping the one `EnemyDef` object a pure, round-independent base definition.
- `Weapon.meleeRange` (checkpoint 16) was chosen as the sole ranged-vs-melee discriminator over adding a separate `kind: "ranged" | "melee"` tag, specifically to avoid two fields that could disagree with each other (e.g. `kind: "melee"` on an object that still has a `magSize` set, or vice versa) — a melee weapon's defining property (having a melee range at all) does double duty as both real data and the type signal. `magSize`/`reloadTime`/`startingReserveAmmo` became optional for the same reason: a shared `Weapon` interface across one `WEAPONS` content array, rather than a parallel type/system for melee, per the project's own "follow existing patterns" instruction for this checkpoint. This change was NOT backward-compatible against the pre-checkpoint-16 `WeaponSystem.ts` at the type-check level (though it was fully runtime-safe) — widening those three fields to optional immediately produced 7 TypeScript errors in the not-yet-rewritten `WeaponSystem.ts`, which is why the interface change and the `WeaponSystem.ts` rewrite were sequenced as adjacent tasks (nothing else landing in between that would need a clean build against the broken intermediate state).
- `WeaponSystem`'s two internal type guards, `assertRangedWeapon()`/`assertMeleeWeapon()` (checkpoint 16, TypeScript assertion functions), exist so the few places that need a `Weapon`'s ammo fields or `meleeRange` guaranteed-defined (constructing a `WeaponSlot`, storing `meleeWeapon`) can narrow the type once, explicitly, with a clear runtime error naming the offending weapon id if the invariant is ever violated — rather than either scattering non-null assertions (`weapon.magSize!`) through `fire()`/`startReload()`/`finishReload()`, or introducing a full discriminated-union split of `Weapon` into `RangedWeapon | MeleeWeapon` (considered, rejected as more invasive than this checkpoint's "adapt minimally" instruction called for, since every existing `Weapon`-typed signature in `MapEntitySystem.ts`/`ui/MainMenu.ts` would have needed to change too).
- **Melee is a quick attack action, never a weapon-equip toggle** (checkpoint 16, corrected mid-checkpoint): the original design had `V` toggle an "equipped" mode (mouse1 swung the knife while equipped; switching guns exited the mode). Manual testing found this was the wrong mechanic entirely, not a tuning issue — real CoD Zombies knife-melee is an instant attack performed while still holding your gun. The corrected `WeaponSystem` never has a `meleeEquipped`-style flag at all: `V`'s own cooldown (`isMeleeAttacking`, derived from a timer comparison, the same pattern gun fire-rate gating already uses) is the only gate, and the equipped gun's slot/ammo/reload state is structurally never touched by any melee code path — there is no "return to the previous gun" step because the gun was never left. This is a stronger invariant than the original design's (which relied on `switchToSlot()` correctly clearing a mode flag); the corrected version has no flag to get out of sync in the first place.
- `meleeWeapon`/`startingMeleeWeapon` are STILL tracked as swappable instance state (checkpoint 16), not a hardcoded constant, even after the equip-toggle redesign above — the reason is unchanged: a planned future Bowie knife would make `V` perform a different, higher-damage attack (see future mechanics), and `meleeWeapon` is what a future reassignment method would target. `reset()` restores it to `startingMeleeWeapon` (captured once at construction) so a future Bowie-knife pickup's effect doesn't persist across a death/respawn. The only thing the equip-toggle correction removed was `meleeEquipped` itself (there is no in-hand "equipped" concept anymore) — the reassignable-attack-type state was never the problem and was kept as-is.
- The melee attack's placeholder feedback (checkpoint 16) reuses `WeaponViewmodel.addImpulse()` (built at checkpoint 14, whose own future-mechanics notes already named melee-swing as an intended integration point) via a new `onMeleeAttack: () => void` constructor callback on `WeaponSystem`, wired in `main.ts` — the same dependency-injection pattern `PlayerState`'s `onDeath` callback already established, so `WeaponSystem` never needs to import `WeaponViewmodel` directly. This required moving `WeaponViewmodel`'s construction earlier in `main.ts`'s `startGame()` (it has no dependency on anything else in that function, so the reorder is side-effect-free) so the callback closure references an already-initialized variable rather than relying on JavaScript's closure-timing semantics to make a forward reference safe.
- **Superseded above** (was: "starts a cooldown (`meleeWeapon.fireRate`, `0.5` seconds)"): the melee attack's cooldown (checkpoint 16, `content/weapons.ts`'s knife `fireRate`) was raised from `0.5` to `1` second after further manual testing found `0.5` let `V` be spammed almost as fast as the gun's own fire cadence, undermining the "distinct action" feel a melee attack is supposed to have. This is purely a value change on existing content — the cooldown *mechanism* (`isMeleeAttacking`, a timer-comparison getter gating both `V` itself and mouse1/`R` for its duration) is exactly the one built during the equip-toggle-to-quick-attack correction above, unchanged.
- The melee attack now plays a dedicated sound, `melee_hit` (checkpoint 16, `content/sounds.ts` + `public/sounds/melee_hit.wav`), instead of reusing `pistol_fire` as the initial implementation did — a low-pitched, fast-decaying synthesized "thud" (Node-generated PCM, matching this project's established placeholder-sound technique), deliberately distinct in character from the gun's sharper click so the two are easy to tell apart by ear. It's non-positional, the same rationale `pistol_fire` already uses (always the local player's own action, no 3D source needed). `WeaponSystem.meleeAttack()` plays it at the same call site as the hit-detection raycast — the same "trigger point" every other damage-dealing sound in this project already fires from.
- **Gun damage now scales per round too — a second, deliberately independent round-scaling coupling alongside the melee-damage-equals-zombie-base-health one above, and the two must never be conflated or "simplified away" into a single formula** (checkpoint 16): `ZombieSurvival.gunDamageMultiplierForRound(round) = 1 + GUN_SCALE_RATE * (round - 1)`, `GUN_SCALE_RATE = 0.3` (a named tunable constant, deliberately gentler than health's per-round growth rate of `1.0`), assigned once per round to a new `WeaponSystem.damageMultiplier` field (default `1`) inside `startRound()`, right alongside the existing `healthForRound()` call. `WeaponSystem.fire()` (guns only) multiplies `weapon.damage` by this field; `meleeAttack()` never reads it, so the knife's damage stays flat at `100` regardless of round, leaving the `damage-100-equals-zombie-base-health` coupling documented above completely untouched. This exists because `healthForRound()`'s linear, uncapped growth alone made gun kill-shot counts grow just as unboundedly (round 10 would need 100 pistol hits), which the project's own future-mechanics notes had left as an open question rather than a settled design; `GUN_SCALE_RATE`'s gentler rate keeps gun difficulty rising with round (not flat, not trivial) while staying survivable at high rounds, rather than forcing melee-only play once guns fall too far behind. `damageMultiplier` was deliberately built as a generic, mode-agnostic `number` field on `WeaponSystem` (not a `ZombieSurvival`-specific concept hardcoded into `core/`), preserving the "`core/` never references `content/` or `modes/`" rule — `WeaponSystem` has no idea rounds exist, it only multiplies whatever the field currently holds, and `ZombieSurvival` (which is allowed to depend on `core/`) is the one place that assigns meaning to it, mirroring the `onMeleeAttack` callback's dependency-injection pattern from the correction above. `ShootingRange` never touches this field, so it stays at its safe default of `1` in that mode — confirmed no scaled-vs-unscaled damage discrepancy exists there. **Anyone touching either `healthForRound()` or `gunDamageMultiplierForRound()` in the future must consider both together**: `healthForRound()` alone determines exactly how many melee hits round N takes (via the fixed 100-damage coupling above); `gunDamageMultiplierForRound()` exists purely to partially compensate the same health growth for guns, at a different rate — changing one without deliberately re-deriving the other breaks either the melee-hits-equal-round-number property or the intended gun-difficulty curve.

- `Campaign.getVaultPin` (checkpoint 19) is deliberately an arrow-function class field (`getVaultPin = (): string => { return this.vaultPin; };`), not a regular method — because `main.ts` passes it around as a bare function reference (`campaign.getVaultPin`, to `MapEntitySystem`'s constructor and to `Terminal`), not wrapped in a closure at each call site (`() => campaign.getVaultPin()`). - The unified two-phase melee swing (checkpoint 21 addendum) replaces the checkpoint-16 single-`addImpulse()` lunge with two calls — a downward dip, then (staggered by a plain `setTimeout(..., 70)`) a right-to-left swing — fired identically on *both* `weaponViewmodel.addImpulse()` and `handsViewmodel.addImpulse()` from the one `onMeleeAttack` callback, unconditionally, regardless of `hasActiveWeapon()`. - `HandsViewmodel`'s knife mesh (checkpoint 21 addendum) is a permanent child of the right hand group only (added after the left-hand `clone()`, specifically so it's never mirrored onto the left hand), built once at construction time with `visible = false`, toggled by a new `setKnifeVisible()` method rather than being constructed/destroyed per swing — matching this project's general preference for toggling `.visible` on a persistent mesh over repeated construction (the same pattern doors, pickups, and the computer-part power cable already use). Two boxes, a thin blade (dark metal tone) and a slightly thicker hilt (a dark near-black grip tone, matching the dark surface colors this project already uses for grip/panel-like parts elsewhere, e.g. `ComputerMesh.ts`'s keyboard), positioned as if gripped in the fist and extending forward in the same local +Z direction the fingers already curl toward.
- **Checkpoint 22 replaces the checkpoint-21-addendum's melee wiring entirely**, rather than extending it, because the addendum's own shape — a shared impulse trick played on whichever viewmodel happened to be active — was structurally incapable of ever showing a real choreographed attack: an impulse only ever nudges an already-visible object, it can't retract one thing and bring in another. This isn't a case of the checkpoint-21-addendum approach being buggy; it did exactly what it was built to do (a quick, cheap swing readable on either carrier). It was simply the wrong mechanism for what checkpoint 22 asked for, so it was removed outright rather than layered under/beside the new sequencer — keeping both would have left two competing melee-feedback paths with no clear owner.
- `MeleeViewmodel`'s stab/swing performances (checkpoint 22) are explicit, driven, timed tweens — not built on `core/utils/ImpulseOffset.ts`, despite `ImpulseOffset` being this project's established "temporary offset" mechanism since checkpoint 14. `ImpulseOffset`'s entire design is "decay linearly back toward the base pose from wherever it currently is, on its own schedule" — correct for a reactive nudge (fire-kick, swap-dip, footstep bob) where the *causing* event is instantaneous and the visual response is a simple decay curve, but wrong for a performance with a specific choreographed start, middle, and end pose (guard → thrust → pull-back for a stab) that another class (`MeleeSequencer`) needs to synchronize its own phase timing against via a known, queryable duration (`getDuration()`). An impulse has no notion of "how long until this settles to a specific pose," only "how long until it decays to zero" — insufficient for a sequencer that needs to know exactly when to start the next phase.
- `MeleeSequencer` (checkpoint 22) reuses `core/utils/StateMachine.ts` rather than a bespoke local timer/phase implementation, since its four phases (idle/retracting/performing/returning), each with its own enter-time setup and per-frame advance logic, is exactly the shape `EnemyAI`'s existing idle/chase/attack usage already proved out — phase *transitions* are still decided by `MeleeSequencer.update()` itself (comparing `phaseElapsed` against each phase's duration), not by the state machine, matching how `EnemyAI` already uses it. This was a judgment call the checkpoint's own spec explicitly left open ("if StateMachine's API doesn't fit... a small local implementation is also fine") — it fit cleanly, so it was reused rather than duplicated.
- **Known, deliberately unhandled edge case**: `MeleeSequencer.trigger()` captures `wasWeaponActive` once, at the instant `V` is pressed, and never re-reads it for the rest of that sequence. If the player switches weapon slots (or a wall-buy replaces the active weapon) *during* an in-progress retract/perform/return, the sequence still hands off to whichever carrier was captured at trigger time, even though `WeaponSystem.hasActiveWeapon()`'s live value could theoretically have changed in the interim (in practice, changing slots mid-sequence doesn't stop the sequence, so this is reachable, just not exercised in this checkpoint's own verification pass). This was flagged in the spec as "out of scope unless it visibly breaks carrier hand-off during testing" — it didn't surface as a visible problem in manual testing, so it was left as documented, not-yet-fixed behavior rather than guarded against.
- `MeleeViewmodel`'s `GUARD_POSITION`/`STAB_THRUST_POSITION` (checkpoint 22) were corrected mid-checkpoint, before shipping, after `THREE.Vector3.project()`-based verification (not just eyeballing a screenshot) showed the first-guess values — modeled loosely after `HandsViewmodel`'s own hand offsets — projected below the visible screen area entirely (NDC y past `-1`) at their chosen depth. The fix shifted both to a shallower `y` and a more negative `z` (farther from the viewmodel camera, which reduces the y-magnitude needed to stay within the vertical frustum at that depth), re-deriving `STAB_THRUST_POSITION` by the same relative delta from the corrected guard pose so the stab's forward-thrust *motion* stayed the one originally designed, just re-based onto a pose that's actually visible. `STAB_PULLBACK_POSITION` was deliberately left unchanged — ending off-screen-low there is the intentional visual handoff to the carrier rising back into view "from below" during `returning`, not the same kind of bug.
- **Superseded at checkpoint 25** (was: "**Superseded at checkpoints 23 and 24** (was: "Reload animation remains not built (it was never in scope for checkpoint 21 or 22) — see the "Future mechanics" section below; it's now explicitly tracked as checkpoint 23 territory rather than an open-ended future item, since fire-kick/swap-dip/melee are now all built and reload is the one remaining `WeaponSystem`-driven viewmodel event with no visual feedback at all."): checkpoint 23 deliberately did not build it either — its own scope was explicitly "just the weapon + its shape," naming reload animation (alongside a support-hand) as follow-on work. Checkpoint 24 built the support hand but, per its own explicit out-of-scope note, left it static once positioned — still not built; checkpoint 25 is now the one that must actually animate it. See the future-mechanics entry below."): built — `core/ReloadSequencer.ts` gives the AK-47 a full multi-phase reload animation, finally moving the support hand and the named magazine/bolt parts checkpoints 23/24 had been building toward. M1911/MAC-10 deliberately did not get the same treatment — see the two new decisions-log entries below for why.
- Tactical-vs-empty reload (checkpoint 25) is decided by a single boolean, `emptyReload`, captured exactly once — inside `WeaponSystem.startReload()`, as `currentAmmo === 0`, before `isReloading` is even set `true` and long before `finishReload()` ever touches ammo — and passed to `ReloadSequencer.trigger()`/the generic-dip branch as a plain argument, never re-derived mid-sequence. This matters because `currentAmmo` itself changes the instant the reload completes (`finishReload()` restores it to a full magazine), so any code that tried to check "was this an empty reload" by re-reading `currentAmmo` partway through the sequence would get the wrong answer once the underlying gameplay timer finished — capturing the boolean once at the moment reload starts, the same way `MeleeSequencer.trigger()` already captures `wasWeaponActive`/`hitEnemy` once rather than re-reading live state later, avoids that entirely.
- Every `ReloadSequencer` phase's duration is a **percentage of `weapon.reloadTime`**, never a hardcoded second value (checkpoint 25) — the same forward-looking reasoning `MeleeSequencer`'s own phase durations already follow (`RETRACT_DURATION`/`RETURN_DURATION` are fixed seconds there only because nothing in this project varies melee speed yet; reload speed, by contrast, is an explicitly named future perk in this project's own future-mechanics notes). Deriving every phase's start/end as a fraction of `weapon.reloadTime` means a future reload-speed perk scales the *entire* choreography — tilt, magazine, bolt, support-hand, and even which sounds play at which relative moment — automatically, just by changing the one number `WeaponSystem`'s existing reload timer already reads, with no changes needed anywhere in `ReloadSequencer.ts` itself.
- M1911/MAC-10 got a simple `ImpulseOffset`-based dip instead of their own `ReloadSequencer`-style choreography (checkpoint 25) for a concrete reason, not a shortcut: `ReloadSequencer`'s entire design depends on named parts to move (`AK47_MAGAZINE_NAME`/`AK47_BOLT_NAME`) and a support hand to hand off between them — neither M1911 nor MAC-10 has a per-weapon mesh at all (both still render `WeaponViewmodel`'s generic box, per checkpoint 23's own deliberate scoping), so there is nothing for a phase sequencer to animate independently. A single `addImpulse({x:0,y:-0.1,z:0}, weapon.reloadTime)` call gets the same "something visibly happens for the whole reload duration" result with zero new mechanism, reusing `ImpulseOffset`'s existing decay-over-time behavior exactly as intended. If M1911/MAC-10 ever get their own per-weapon meshes (see the still-open "M1911/MAC-10 own viewmodel meshes" future-mechanics entry), a real phase sequence for them would become possible for the first time, but isn't built now because there's nothing yet for it to move.
- `WeaponMesh.ts`'s three named parts (checkpoint 23, `AK47_BODY_NAME`/`AK47_MAGAZINE_NAME`/`AK47_BOLT_NAME`) exist with no consumer yet — nothing in this checkpoint calls `getObjectByName()` on them. They were named anyway, unlike the barrel/handguard, specifically because a reload animation (the checkpoint immediately deferred to, see above) is expected to need to move exactly these three parts independently (eject/insert the magazine, rack the charging handle, keep the body/receiver as the stable parent everything else is positioned relative to) — the same "name only what a caller will actually look up" restraint `ComputerMesh.ts`'s `COMPUTER_BODY_NAME` already established, just applied slightly ahead of the checkpoint that will actually call `getObjectByName()` on them, because the shape of that future need was already concrete enough to name correctly now rather than guess wrong later.
- `WeaponViewmodel`'s `displayGroup` wrapper (checkpoint 23) — both `genericMesh` and `ak47Mesh` are built once at construction and toggled via `.visible`, never added/removed/rebuilt on a weapon switch. This was a deliberate choice over lazily constructing the AK-47 mesh on first equip: both are cheap procedural-box geometry (no textures, no async loading), so there's no real cost to building both up front, and toggling a boolean is strictly simpler and cheaper per-switch than construct/dispose bookkeeping — the same "toggle `.visible` on a persistent mesh" preference this project has followed since doors/pickups (checkpoint 6) and reused explicitly for `HandsViewmodel`'s now-removed knife (checkpoint 21 addendum) and `MeleeSequencer`'s carrier hand-off (checkpoint 22).
- **Superseded at the checkpoint-23 fix** (was: "The AK-47 mesh and the generic box occupy the mathematically identical position within `displayGroup` (checkpoint 23) — both sit at local `(0,0,0)`, with `displayGroup` itself carrying the entire base+bob+impulse+sequencer offset that used to be applied directly to the single `weaponMesh`. This was a deliberate design constraint, not an accident: the checkpoint's own spec required switching between AK-47 and the other weapons not to "jump wildly in scale," and guaranteeing both meshes share one parent transform is what makes that true by construction rather than by matching two independently-tuned offsets by eye. Confirmed via `THREE.Vector3.project()` against `WeaponViewmodel`'s own camera, the same verification technique `MeleeViewmodel`'s checkpoint-22 guard-pose fix already established for this project."): sharing the exact same position turned out to be the wrong goal once the AK-47's real, longer silhouette existed to actually look at — it left far too little of the rifle on screen. A same-checkpoint fix gave `ak47Mesh` its own `AK47_OFFSET`, applied only to that mesh's local position (not to `VIEWMODEL_CONFIG`, which the generic box still reads), moving it up and left. The underlying *mechanism* this bullet described — one shared parent (`displayGroup`) carrying the base+bob+impulse+sequencer math, with each child mesh only needing its own static local offset on top — is unchanged and is exactly what checkpoint 24's support-hand mesh also relies on; only the specific claim that AK-47 and the generic box share the *same* local offset is no longer true.
- `WeaponViewmodel`'s support-hand mesh (checkpoint 24) reuses `utils/HandMesh.ts`'s `createFistMesh()` — the same closed-fist shape `MeleeViewmodel` already uses — rather than the open-palm `createHandMesh()` `HandsViewmodel` uses for idle hands. A gripping fist around a handguard is the correct pose for this purpose; an open flat palm (designed to read as an empty, relaxed hand) would look wrong wrapped around a foregrip. This is the second consumer of `createFistMesh()` after `MeleeViewmodel`, confirming the checkpoint-22 extraction of that factory into `core/utils/` (rather than leaving it inline in `MeleeViewmodel.ts`) was the right call ahead of this exact kind of reuse.
- The support hand inherits `WeaponViewmodel`'s existing walk-bob/impulse/sequencer motion by being a `displayGroup` child, not by any new code written for it (checkpoint 24). This was confirmed, not assumed: `displayGroup`'s own position was checked before and after a fire-kick impulse and found to change, which — given Three.js's parent-child transform composition is unconditional — proves any child of `displayGroup`, support hand included, moves with it automatically. This mirrors the exact reasoning `ak47Mesh` already relied on at checkpoint 23 (see the entry above) and is the reason this checkpoint's own spec correctly anticipated "zero new per-frame code" as achievable rather than aspirational.
- The support hand's visibility/position update is folded into the *same* `weaponId !== lastWeaponId` check the checkpoint-23 AK-47 mesh swap already added (checkpoint 24), not a second parallel check keyed off something else. Both are, structurally, "does this aspect of the viewmodel's appearance need to change for the newly-equipped weapon" — keeping them in one check means there's only one place `WeaponViewmodel.update()` ever reads `activeWeapon` to decide appearance, not two that could drift out of sync (e.g. one checking `weaponId` directly, another re-deriving it differently).

## Future mechanics (documented, not built)

Ideas that came up while implementing checkpoint 4.5 but are deliberately out of scope until a later checkpoint gives them a real home. Listed so they're designed-for-later, not forgotten.

- **Downed/revive state for the player**: a third `playerState` value (e.g. `"downed"`) between `"alive"` and `"dead"`, gated behind a perk that doesn't exist yet. The string-union type was chosen specifically so adding this later doesn't require touching every call site that currently checks `=== "alive"` as a boolean-like condition.
- **Equivalent downed ("crawler") state for enemies**: an enemy that, below some health threshold, becomes slower and weaker instead of dying outright — reusing the same state-union pattern as the player's.
- **Perk system framework**: not designed yet. The downed/revive mechanic above depends on it existing first.
- **Post-death flow beyond "YOU DIED" + Respawn/Main Menu**: a "survived N rounds" summary is now shown (checkpoint 7); a full spectate mode is still not built.
